1 什么是 Agent Teams

多个 AI Agent 组成团队，按角色分工、并行协作、自动编排

Agent Teams 核心概念
角色分离
Spec (Opus) — 只写规范，不写代码
Coder (Sonnet) — 严格按规范实现
Reviewer — 审查 + 反馈循环
Integrator — 端到端集成验证
并行执行
多个 Coder 操作不同目录
Task 依赖自动阻塞/解除
不同包路径避免文件冲突
完成后自动通知下游
质量保障
规范驱动 — 先设计后编码
每个模块独立单元测试
集成测试验证协作正确性
合规约束自动注入 prompt
自动编排
Team Lead 协调任务流转
TaskCreate + 依赖关系
SendMessage 跨 Agent 通信
完成后自动 shutdown 清理

核心理念

传统模式：一个 AI 串行完成所有工作，上下文膨胀，容易出错。
Agent Teams：拆解 → 分工 → 并行 → 集成，每个 Agent 专注一件事，上下文精简，质量更高。

5 实战经验总结

踩坑与最佳实践

实战经验
成功要素
Spec 先行 — 规范质量决定代码质量，用 Opus 不省
目录隔离 — 每个 Coder 独占包路径，零冲突
约束注入 — CLAUDE.md 的禁止项自动进入每个 Agent
测试驱动 — 每个模块独立测试 + 集成测试兜底
注意事项
跨包修改 — Coder 可能"好心"修改别人的文件
Spec 偏差 — 规范中的矛盾需要 Team Lead 仲裁
Shutdown 延迟 — 有些 Agent 需要多次 shutdown 请求
依赖冲突 — 多个 Coder 同时修改 pom.xml / yml
最佳实践
Opus for thinking, Sonnet for doing
Task 依赖 — blockedBy 确保正确执行顺序
明确边界 — 告诉每个 Agent "你只操作 X 目录"
全量验证 — 最后 mvn test 全量跑一次
维度 单 Agent 模式 Agent Teams 模式
上下文窗口 全部堆在一个窗口，容易溢出 每个 Agent 独立上下文，精简聚焦
并行度 串行，一个做完才能做下一个 多模块真正并行执行
质量保障 自己写自己查，容易遗漏 规范驱动 + 交叉验证 + 集成测试
一致性 上下文长了容易"忘记"约束 约束在每个 Agent prompt 中独立注入
复杂度 简单任务更高效 3+ 模块时优势明显
