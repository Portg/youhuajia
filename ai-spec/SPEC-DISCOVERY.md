# SPEC-DISCOVERY.md — SPEC 四阶段对话生成流程

> 在创建或修改 `intent.md` 前使用本流程。
> AI 通过结构化对话暴露业务边界，而非直接编码。

---

## Phase 1：业务追问（10 轮深挖）

**AI 角色**：业务分析师

目标：理解「为什么做这个功能」而非「怎么做」

### 追问模板

```
你即将为「优化家」项目定义一个新的业务需求。
请回答以下问题（每个问题 1-3 句话即可）：

1. 这个功能解决用户的什么痛点？
2. 用户在没有这个功能时怎么做？
3. 成功标准是什么？用户做了什么算「完成」？
4. 这个功能失败了会怎样？最坏情况？
5. 谁是这个功能的主要用户？他们的特征？
6. 这个功能和现有哪些功能有依赖关系？
7. 数据从哪来？（用户手动输入 / OCR / 第三方接口）
8. 数据精度要求？（展示用 / 计算用 / 存储用）
9. 并发场景？（一个用户多设备 / 多个用户同时操作）
10. MVP 阶段做到什么程度就够了？
```

---

## Phase 2：破坏者模式（挖 Anti-Goals）

**AI 角色**：恶意用户 / 攻击者

目标：找出「不应该发生但可能发生」的边界情况

### 追问模板

```
现在我扮演破坏者，尝试滥用这个功能。请告诉我以下场景是否应该被阻止：

1. 如果我输入一笔 0 元的债务？
2. 如果我输入 APR 为 999% 的债务？
3. 如果我同时从两个设备提交同一笔债务？
4. 如果我在 OCR 结果未确认时就请求生成报告？
5. 如果我在 1 分钟内连续请求 100 次计算？
6. 如果我伪造 requestId 重复提交？
7. 如果我直接调用内部接口绕过状态机？
8. 如果我的月收入填 0 但有 10 笔债务？
9. 如果 OCR 识别出的金额和我手动输入的不一致？
10. 如果我删除一笔已经纳入画像计算的债务？
```

输出：每个场景的处理策略（阻止 / 告警 / 允许 + 原因）

---

## Phase 3：合规审计（数据敏感级别）

**AI 角色**：合规审计员

目标：标定每个字段的敏感级别和处理要求

### 审查模板

```
请逐个审查本功能涉及的数据字段：

对每个字段回答：
1. 敏感级别：高敏 / 中敏 / 低敏
2. 日志是否可以输出？如何脱敏？
3. 响应是否返回前端？如何脱敏？
4. 存储是否需要加密？
5. 是否涉及个人金融信息（PII/PFI）？
6. 数据保留期限？
```

---

## Phase 4：输出 intent.md

基于 Phase 1-3 的对话结果，AI 起草 `intent.md`，包含：

1. **核心业务意图**（1-3 句话）
2. **Anti-Goals 表**（编号 + 描述 + 违反后果 + 对应测试方法名）
3. **数据敏感级别表**
4. **业务边界**（做什么 / 不做什么）
5. **用户分群策略**（如适用）

人工审定后合入 `ai-spec/domain/intent.md`。

---

## 使用时机

| 场景 | 是否使用 SPEC-DISCOVERY |
|------|------------------------|
| 新增核心功能模块 | 必须使用全部 4 个 Phase |
| 修改现有功能的业务规则 | Phase 2 + Phase 3 |
| 纯技术重构（不改业务逻辑） | 不需要 |
| 新增 API 端点 | Phase 1 + Phase 2 |
| 修改 AI Prompt | 不需要（走 Prompt 版本管理） |
