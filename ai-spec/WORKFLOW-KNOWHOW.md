# WORKFLOW-KNOWHOW.md — 迭代方式与 ai-spec 使用约定

> 本文档约定：在 Vibe Coding 下，**小功能迭代**与**大规模迭代**分别如何配合 ai-spec 使用，以及何时「先改 spec」、何时「直接改代码」。

---

## 一、核心结论

| 迭代类型     | 推荐方式                     | 说明 |
|--------------|------------------------------|------|
| **小功能迭代** | **直接处理**为主             | 在对话里按需 @ 相关 .md，不要求先改 spec；只有动到「契约」时才先改 spec。 |
| **大规模迭代** | **先通过 ai-spec 文件**      | 先更新对应 spec（接口/实体/状态机/规则/用户路径等），再按 spec 实现或重构，保证单源真相。 |

---

## 二、何时「直接改代码」（小迭代）

- 文案、样式、小交互、修 bug。
- 在**现有** API 内加可选字段、现有页面内加逻辑，且不改变 URL/请求体结构/表结构。
- 配置、依赖、测试用例补充、日志与可观测性。

**操作建议**：直接描述需求或贴代码片段，在 prompt 里注明「遵循 CLAUDE.md / user-journey / error-codes」等即可，无需先改 spec。若改完后发现某份 spec 与实现不一致，再**反向更新**该 spec（文档追代码）。

---

## 三、何时「先改 ai-spec 再投喂」（契约级 / 大规模）

以下变更会改变**约定**，建议先改对应 spec，再让 AI 或人工按 spec 实现：

| 变更类型       | 建议先改的 ai-spec 文件 |
|----------------|--------------------------|
| 新增/修改 API  | `contracts/openapi.yaml` |
| 错误码变更     | `contracts/error-codes.md` |
| 实体/表结构   | `domain/entities.md` + Flyway 迁移 |
| 状态流转       | `domain/state-machines.yaml` |
| 用户路径/漏斗 | `domain/user-journey.md` |
| 评分/规则/引擎 | `engine/*.md`、`strategies/*.yml` |
| 客户端结构/页面 | `client-spec.md`、`pages.json` 等 |
| 测试矩阵/数据 | `test/test-matrix.md`、`test/mock-data.md` |

**操作建议**：只改与本次迭代相关的 1～2 个 spec 文件，改完后在 prompt 中明确：「按 ai-spec/xxx.md 的更新实现/修改 yyy」。

---

## 四、判断口诀

- **只动实现、不动约定** → 直接改代码。
- **动接口、动表、动流程、动规则** → 先改 ai-spec，再按 spec 改代码。

---

## 五、与现有文档的关系

- **CLAUDE.md**：每次生成/修改代码都应遵循的全局约束；小迭代时在对话里 @ 即可。
- **README.md / 文档索引**：说明「何时读取」哪份 spec；大规模迭代时按需更新对应 spec 再按「何时读取」投喂。
- 本文件：**迭代方式**的 knowhow，不替代上述规范，只约定「先改 spec 还是直接改代码」的用法。

---

## 六、维护建议

- ai-spec 用 Git 管理，有变更就 commit，便于回溯「约定何时改的」。
- 直接改代码后若发现与某 spec 不一致，尽量**补一次 spec 更新**，保持实现与文档一致。
