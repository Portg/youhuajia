# /debug — 通用问题排查技能包

当用户输入 `/debug` 后跟报错信息或问题描述时，按照系统化流程进行问题排查。

## 触发方式

```
/debug <报错信息 | 问题描述 | 图片路径 | 日志文件路径>
```

示例：
- `/debug Uncaught TypeError: canvas.getContext is not a function`
- `/debug 页面白屏，控制台无报错`
- `/debug 接口返回 500，日志无异常`
- `/debug 编译警告 Sass @import deprecation`
- `/debug 内存溢出 OOM`
- `/debug ~/Downloads/20260305-101033.jpg`
- `/debug /tmp/app-error.log`

---

## 执行流程

### 第零步：输入识别与信息采集

根据用户输入的类型，自动识别并提取有效排查信息：

| 输入类型 | 识别方式 | 处理方法 |
|---------|---------|---------|
| **报错文本** | 包含 `Error`、`Exception`、状态码等关键字 | 直接解析，提取错误类型、文件、行号、堆栈 |
| **截图/图片** | 路径以 `.jpg`、`.png`、`.jpeg`、`.gif`、`.webp`、`.bmp` 结尾 | 用 Read 工具查看图片 → 识别报错内容 → 提取关键信息 |
| **日志文件** | 路径以 `.log`、`.txt`、`.out` 结尾，或包含 `/logs/`、`/tmp/` | 用 Read 工具读取 → 搜索 ERROR/WARN/Exception 段落 → 提取关键信息 |
| **模糊描述** | 无具体报错，如"打不开""不对""卡了" | 进入追问模式（见下方） |

**截图处理流程**：
1. 用 Read 工具打开图片文件（Claude 支持多模态识别）
2. 从图片中提取：错误类型、报错信息、文件路径、行号、堆栈片段
3. 如果图片模糊或信息不全，告知用户缺少什么信息并请求补充
4. 将提取的文本信息作为输入，进入第一步问题分类

**模糊描述追问模式**：
当用户输入信息不足以定位问题时，用 AskUserQuestion 追问（最多 2 轮）：
- **现象**：具体什么行为不符合预期？
- **时机**：什么时候开始出现的？之前正常吗？
- **范围**：所有页面/接口都有问题，还是特定场景？
- **环境**：在什么环境下（浏览器/小程序/真机）？
- **复现**：每次都能复现还是偶发？

追问后仍信息不足时，主动采集：
1. 运行构建命令查看编译错误
2. 检查 `git log --oneline -10` 最近变更
3. 搜索项目中的 TODO/FIXME/HACK 标记

---

### 第一步：问题分类

根据报错信息自动判断问题类型：

| 类型 | 识别特征 | 典型场景 |
|------|----------|----------|
| 编译/构建错误 | `Error`, `Build failed`, `Cannot find`, `Undefined variable` | 依赖缺失、语法错误、配置错误 |
| 运行时异常 | `TypeError`, `ReferenceError`, `NullPointer`, `ClassCast` | 类型不匹配、空值、API 误用 |
| 网络/接口错误 | `4xx`, `5xx`, `CORS`, `timeout`, `ECONNREFUSED` | 接口不通、跨域、超时 |
| 渲染/UI 异常 | 白屏、样式错乱、组件不显示、闪烁 | CSS 冲突、生命周期、平台差异 |
| 性能问题 | 卡顿、OOM、慢查询、内存泄漏 | N+1 查询、大对象、未释放资源 |
| 数据问题 | 数据不一致、精度丢失、乱码 | 编码、浮点精度、时区、并发 |
| 环境/配置问题 | 本地正常线上异常、版本不一致 | 环境变量、依赖版本、平台差异 |

### 第二步：探测项目技术栈

读取当前项目根目录下的 `CLAUDE.md`，提取：
- 技术栈（语言、框架、数据库、ORM 等）
- 禁止事项（避免排查方向违反项目约束）
- 强制约束（排查时需考虑的特殊规则）

如果没有 `CLAUDE.md`，则根据项目文件自动检测：

| 检测文件 | 技术栈 |
|----------|--------|
| `pom.xml` | Java / Maven / Spring |
| `build.gradle` | Java / Gradle |
| `package.json` | Node / 前端框架 |
| `pubspec.yaml` | Flutter / Dart |
| `go.mod` | Go |
| `requirements.txt` / `pyproject.toml` | Python |
| `Cargo.toml` | Rust |
| `composer.json` | PHP |

同时检测前端框架：`uni-app`、`React`、`Vue`、`Angular`、`Flutter` 等。

### 第三步：三级排查（逐级深入）

根据问题复杂度逐级排查，简单问题在 L1 就能解决，无需深入。

---

#### L1 初级排查（基础检查，覆盖 70% 的问题）

**目标**：排除最常见的低级错误，3 分钟内给出结论。

**通用检查清单**：
1. **完整阅读报错信息** — 提取关键字段：错误类型、文件路径、行号、堆栈
2. **精确定位源文件** — 用 Grep/Glob 找到报错文件和行号，Read 读取上下文
3. **检查拼写和语法** — 变量名、导入路径、括号配对、大小写
4. **检查依赖是否存在** — 导入的模块/类/函数是否确实存在
5. **检查版本兼容性** — API 是否在当前版本可用，是否已废弃
6. **搜索项目同类用法** — Grep 搜索项目内其他文件是怎么用的，对比差异

**技术栈适配检查**（根据第二步探测结果自动选择）：

<details>
<summary>前端（Vue / React / uni-app / 小程序）</summary>

- 浏览器控制台完整报错信息
- 组件是否正确注册/导入
- 生命周期时机是否正确（DOM 是否已挂载）
- **跨平台差异**：H5 vs 小程序 vs App 的 API 差异
- CSS 作用域：scoped 样式、样式变量是否可达
- 条件编译：`#ifdef` 分支是否正确
</details>

<details>
<summary>后端（Java / Spring / Node / Go / Python）</summary>

- 完整堆栈日志（不只看第一行）
- 依赖注入是否成功（Bean 是否注册）
- 配置文件是否生效（profile、环境变量）
- 数据库连接是否正常
- 事务边界是否正确
</details>

<details>
<summary>构建工具（Maven / Gradle / Webpack / Vite）</summary>

- 依赖版本冲突
- 插件配置语法
- 缓存导致的残留问题
- 环境变量 / PATH 是否正确
</details>

**L1 输出格式**：
```
## L1 排查结果

**问题类型**：[编译错误 / 运行时异常 / ...]
**根因定位**：[具体原因]
**修复方案**：[具体步骤]
**关联文件**：[文件:行号]
```

如果 L1 能定位问题，直接输出修复方案。否则进入 L2。

---

#### L2 中级排查（系统分析，覆盖 25% 的问题）

**目标**：通过上下文分析和关联排查定位问题，10 分钟内给出结论。

**通用排查方法**：
1. **调用链追踪** — 从报错点逆向追踪完整调用链，找到真正的源头
2. **数据流分析** — 跟踪数据从输入到报错点的完整流转过程
3. **对比分析** — 找到项目中正常工作的类似功能，逐项对比差异
4. **最近变更排查** — `git log` / `git diff` 查看最近改动，重点关注报错文件及其依赖
5. **环境差异排查** — 对比不同环境（dev/prod、H5/小程序）的行为差异
6. **隔离测试** — 最小化复现问题，排除干扰因素

**技术栈深度检查**：

<details>
<summary>前端深度排查</summary>

- 响应式数据是否正确触发更新（Vue: ref/reactive、React: state/effect）
- 异步时序问题（await 遗漏、竞态条件）
- 第三方库版本与框架版本兼容性
- 打包产物分析（source map 映射到源码）
- 平台 API 包装层差异（uni-app 的 canvas/storage/request 在不同平台的实现差异）
</details>

<details>
<summary>后端深度排查</summary>

- SQL 执行计划分析
- 线程/协程状态检查
- 内存 dump / GC 日志分析
- 中间件（Redis/MQ）连接状态
- AOP/拦截器/过滤器的执行顺序
</details>

**L2 输出格式**：
```
## L2 排查结果

**调用链**：[A → B → C → 报错点]
**数据流**：[输入值 → 中间状态 → 异常状态]
**根因定位**：[深层原因]
**修复方案**：[具体步骤 + 代码示例]
**影响范围**：[哪些功能可能受影响]
**回归验证**：[如何验证修复有效]
```

如果 L2 能定位问题，输出修复方案。否则进入 L3。

---

#### L3 高级排查（深度诊断，覆盖 5% 的疑难问题）

**目标**：解决复杂的跨层、跨系统、间歇性问题。

**通用诊断方法**：
1. **全链路日志** — 从入口到出口的完整日志串联（traceId）
2. **二分法定位** — 回退 commit / 注释代码块，缩小范围
3. **源码级分析** — 阅读框架/库源码，理解内部行为
4. **并发/时序分析** — 多线程竞争、事件循环阻塞、死锁检测
5. **底层机制排查** — 内存布局、编码转换、序列化/反序列化
6. **社区/Issue 搜索** — 在 GitHub Issues、Stack Overflow 搜索同类问题

**L3 输出格式**：
```
## L3 排查结果

**问题复现条件**：[精确的复现路径]
**根因分析**：[框架/平台/底层原因]
**修复方案**：[方案 A / 方案 B，对比优劣]
**临时规避**：[如果修复需要时间，临时方案]
**防御措施**：[如何避免类似问题再次发生]
**参考资料**：[相关文档/Issue 链接]
```

---

### 第四步：执行修复与验证闭环

排查定位完成后，进入修复阶段。**必须征得用户同意后再执行修改。**

#### 4.1 修复前确认

向用户展示修复计划，使用 AskUserQuestion 确认：

```
## 修复计划

**根因**：[一句话总结]
**修改文件**：
- `文件A:行号` — 改动说明
- `文件B:行号` — 改动说明

**修改范围**：[最小 / 中等 / 较大]
**风险评估**：[低 / 中 / 高]
```

选项：
- 执行修复
- 只看方案，我自己改
- 需要调整方案

#### 4.2 执行修复

按以下顺序执行：

1. **修改代码** — 使用 Edit 工具，最小改动，不借机重构
2. **同步修改** — 如果同一逻辑存在多处副本（如 `pages/` 和 `src/pages/` 的镜像文件），必须同步修改
3. **关联修复** — 修复引发的级联问题（如重命名变量后更新所有引用）

**修复原则**：
- 只改导致问题的代码，不改周边"顺便优化"
- 遵守项目 `CLAUDE.md` 中的所有禁止事项和强制约束
- 保持与项目现有代码风格一致

#### 4.3 构建验证

修复后立即验证：

**前端项目**：
```bash
# 编译检查
npx vite build 2>&1 | grep -iE "(error|fail)" | head -10
# 或 npm run build
```

**后端项目**：
```bash
# 编译 + 测试
mvn compile 2>&1 | tail -5
mvn test 2>&1 | tail -20
# 或 ./gradlew build / go build / cargo build
```

**通用检查**：
- 编译是否通过（零 error）
- 相关测试是否通过
- 是否引入新的警告

#### 4.4 回归检查

确认修复没有引入新问题：

1. **Grep 扫描** — 搜索项目中是否还有相同模式的问题
   ```
   Grep 搜索与本次 bug 相同的错误模式，确认没有遗漏
   ```
2. **关联文件检查** — 使用相同 API/模式的其他文件是否也需要修复
3. **边界场景** — 修复是否覆盖了 null、空值、极端输入等边界

#### 4.5 修复报告

输出最终报告：

```
## 修复完成

**问题**：[一句话描述]
**根因**：[一句话原因]
**修复内容**：
- `文件A:行号` — 具体改动
- `文件B:行号` — 具体改动

**验证结果**：
- [x] 编译通过
- [x] 相关测试通过
- [x] 回归扫描无遗漏

**防御建议**：[如何避免类似问题再次发生，可选]
```

#### 4.6 经验沉淀（可选）

如果问题具有复用价值（跨平台差异、框架陷阱、项目特有约束），写入 memory：
- 问题模式（什么场景下会出现）
- 排查路径（怎么快速定位）
- 修复方案（标准解法）

---

## 排查原则

1. **先读后改** — 必须先 Read 报错文件，理解上下文，再提方案
2. **最小改动** — 修复只改必要的代码，不借机重构
3. **验证闭环** — 每个修复方案必须给出验证步骤
4. **不猜测** — 不确定的问题用工具验证，不凭印象判断
5. **尊重约束** — 修复方案不得违反 `CLAUDE.md` 中的禁止事项
6. **记录沉淀** — 有价值的排查经验写入 memory 供后续复用

## 常见问题速查索引

### 编译类
| 关键字 | 常见原因 | 快速修复 |
|--------|----------|----------|
| `Undefined variable` | SCSS 变量未导入或已重命名 | 检查 `@use` 导入路径和变量名 |
| `Cannot find module` | 路径错误或依赖未安装 | 检查路径大小写、运行 install |
| `SyntaxError` | 语法错误或版本不支持的语法 | 检查语法和目标版本配置 |

### 运行时类
| 关键字 | 常见原因 | 快速修复 |
|--------|----------|----------|
| `is not a function` | 对象类型不对、API 不存在于当前平台 | 检查对象实际类型和平台 API 差异 |
| `Cannot read properties of null` | 未初始化、异步时序、DOM 未挂载 | 加空值检查、调整生命周期时机 |
| `NullPointerException` | 未注入、查询返回 null | 检查依赖注入和 null 处理 |

### 网络类
| 关键字 | 常见原因 | 快速修复 |
|--------|----------|----------|
| `CORS` | 跨域未配置 | 后端添加 CORS 配置或用代理 |
| `404` | 路径错误或资源不存在 | 核对 URL 路径和路由配置 |
| `500` | 后端未捕获异常 | 查看服务端日志的完整堆栈 |

### 渲染类
| 关键字 | 常见原因 | 快速修复 |
|--------|----------|----------|
| 白屏 | JS 报错阻断渲染、路由配置错误 | 查控制台报错、检查入口配置 |
| 样式不生效 | scoped 隔离、优先级不够、变量不可达 | 检查选择器权重和变量导入 |
| 组件不显示 | 未注册、条件渲染逻辑错误、v-if 数据 | 检查 import 和数据初始值 |
