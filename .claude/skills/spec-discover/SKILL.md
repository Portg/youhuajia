# /spec-discover — SPEC 四阶段对话生成

通过结构化对话流程生成或更新 `ai-spec/domain/intent.md`。

## 触发方式

```
/spec-discover <新功能描述 | 已有功能修改描述>
```

示例：
- `/spec-discover 新增债务一键导入功能`
- `/spec-discover 修改评分引擎权重规则`

---

## 执行流程

### 第一步：判断范围

| 场景 | 执行范围 |
|------|----------|
| 新增核心功能模块 | Phase 1 + 2 + 3 + 4 全部 |
| 修改现有功能的业务规则 | Phase 2 + 3 |
| 新增 API 端点 | Phase 1 + 2 |
| 纯技术重构 | 不需要，告知用户直接开发 |

### Phase 1：业务追问（10 轮深挖）

**AI 角色**：业务分析师

依次向用户追问（用 AskUserQuestion，每次 2-3 个问题，最多 4 轮）：

1. 这个功能解决用户的什么痛点？
2. 用户在没有这个功能时怎么做？
3. 成功标准是什么？
4. 这个功能失败了会怎样？最坏情况？
5. 谁是这个功能的主要用户？
6. 和现有哪些功能有依赖关系？
7. 数据从哪来？
8. 数据精度要求？
9. 并发场景？
10. MVP 阶段做到什么程度就够了？

### Phase 2：破坏者模式（挖 Anti-Goals）

**AI 角色**：恶意用户 / 攻击者

读取现有 `intent.md` 的 Anti-Goals，然后针对新功能提出破坏场景：

- 极端输入（0、负数、超大值）
- 并发操作（多设备、多用户）
- 流程绕过（跳过中间步骤）
- 频率攻击（短时间大量请求）
- 数据不一致（OCR 结果与手动输入矛盾）

向用户展示每个场景，确认处理策略（阻止 / 告警 / 允许）。

### Phase 3：合规审计（数据敏感级别）

**AI 角色**：合规审计员

读取 `intent.md` 的数据敏感级别表，对新功能涉及的字段逐个审查：

| 审查项 | 选项 |
|--------|------|
| 敏感级别 | 高敏 / 中敏 / 低敏 |
| 日志处理 | 禁止输出 / 脱敏输出 / 正常输出 |
| 响应处理 | 不返回 / 脱敏返回 / 正常返回 |
| 存储加密 | 是 / 否 |

### Phase 4：输出 intent.md 更新

基于 Phase 1-3 对话结果，生成 `intent.md` 的增量更新：

1. 新增的 Anti-Goals（编号续接现有最大编号）
2. 新增的数据敏感字段
3. 业务边界更新（做什么 / 不做什么）
4. 用户分群更新（如适用）

向用户展示更新内容，确认后写入 `ai-spec/domain/intent.md`。

### 最终输出

```
## SPEC-DISCOVERY 完成

**功能**：{功能描述}
**执行范围**：Phase {1-4}
**新增 Anti-Goals**：{数量}
**新增敏感字段**：{数量}
**intent.md 已更新**：✓

下一步建议：
- 使用 CONTEXT-SLICE 切片 {X} 开始开发
- 使用 PROMPT-VERSIONS/{模板} 生成代码
```
